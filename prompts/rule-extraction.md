# Rule Extraction Prompt

## 使い方
1. このファイルを開く
2. 下のプロンプトをコピー
3. Cursorに貼る
4. 参照するstockファイルを指定する

---

## Prompt
あなたは、Web制作における実務ナレッジを「共通ルール」に変換する専門家です。

以下の stock ファイル群を読み取り、
複数案件に共通して使える内容だけを抽出して、
`coding-rules` 用の md ファイルを作成してください。

---

# 目的

- 過去案件の stock に蓄積された知見から、再利用可能な共通ルールを作る
- 案件固有の事情ではなく、複数案件に通用する実装ルールを残す
- AIや自分が後から参照しやすい形に整理する

---

# 対象

- 入力として渡された複数の stock ファイル
- EJSテンプレート案件
- WordPressテンプレート案件
- 静的コーディング / Vite / PHP / SCSS / JS を含むWeb制作案件

---

# 抽出ルール

以下の基準で抽出してください。

## 抽出するもの
- 複数案件で共通して出てきた判断
- 他案件でも再利用できる実装ルール
- AIが迷いやすいポイント
- 保守性・再利用性・アクセシビリティ・パフォーマンスに関わるルール
- 実装時の失敗やハマりどころから得られた教訓

## 抽出しないもの
- その案件だけでしか成立しない事情
- 固有のクライアント事情
- 単発の例外対応
- 単なる作業ログ
- 再利用性の低い細かすぎるメモ

---

# ルール化の方針

- stock の内容をそのまま並べ替えるのではなく、共通する考え方にまとめる
- 似た内容は統合する
- 曖昧な表現は減らし、判断しやすい言葉にする
- 可能な限り「結論 → 理由 → 例」の順で書く
- 完璧に網羅しなくてよい。まずは重要なものを優先する

---

# 出力方針

- 出力先は `coding-rules/` 配下を想定する
- 内容に応じて、以下のどれか1つを選んで出力する
  - `coding-rules/common.md`
  - `coding-rules/video.md`
  - `coding-rules/wordpress.md`
  - `coding-rules/css.md`
  - `coding-rules/performance.md`

※ どのファイルにすべきか最初に判断してください  
※ まだ分類しきれない場合は `coding-rules/common.md` にまとめてください

---

# 出力フォーマット

ファイル名: {出力先ファイル名}

# {ルール名}

## 結論
- まず守るべきルールを簡潔に書く

## 理由
- なぜそのルールが必要かを書く
- どんな問題を防ぐかを書く

## 具体例
- コード例、実装例、判断例などを書く
- 必要なら EJS / WordPress の違いも書く

## 例外・補足
- 常にこのルールを適用しなくてよい場合
- 案件に応じて調整が必要な条件

## 関連トピック
- 関連するテーマがあれば列挙する
- 例: video / WordPress / EJS / パフォーマンス / アクセシビリティ

---

# 出力時の注意

- 「stock の要約」ではなく「共通ルールの作成」を行うこと
- 案件名は必要最小限にとどめる
- ルール本文は、他案件でもそのまま読める形にする
- 1つのファイルに詰め込みすぎない
- ただし今回はスモールスタート前提なので、迷ったら `common.md` に寄せてよい

---
