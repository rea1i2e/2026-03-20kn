# コーディングルール（このリポジトリ用）

このリポジトリでEJS/HTML・Sass（SCSS）・JavaScript・PHPを実装/改修する際のルールをまとめます。

## 厳守ルール（文言・データ分離）

1. `php` や `ejs`（`html` を生成するテンプレート）で、**HTMLタグ内のテキストコンテンツを勝手に作らない**でください。  
   - デザイン/要件にある文言のみを使用してください。
2. **データ取得とHTML記述は可能な限り分けてください**。  
   - EJSなら `<% ... %>` ブロックで必要なデータ（配列/オブジェクト等）を準備し、描画はその後に書く。
   - `data` を作った直後にHTML側で `<%- ... %>`/`<%= ... %>` を参照する形を基本にする。
3. `php` や `html` の**コンテンツ部分は自動補完しない**でください。  
   - 例：見出し/段落/ボタン文字などは、必ず設計の文言（または既存実装からの参照）を入れる。
4. PHPの配列は、**`array()`ではなく `[]`** を使用してください。  
   - 例外：外部配布/既存資産（例：`mail.php`のような既製スクリプト）をそのまま流用している箇所は、改修時のみローカル方針に寄せる。

## EJS/HTML（構造）

### 1. エントリHTML（`src/**/*.html`）
- 各ページ（例：`src/index.html`）の先頭で `getPage()` により `page` を取得し、以降のincludeに渡します。
- 本ページHTMLは、レイアウト（head/header/footer）と、ページ固有のincludeで構成します。

根拠：
- `src/index.html`  
- `src/contact/thanks.html`

### 2. includeの基本
- 共通部品は `src/ejs/common` / `src/ejs/components` から `include` します。
- `ejsPath` は `config/site.config.js` が提供します。ページ側では必ず `ejsPath` を使って参照してください。
  - 例：`include(ejsPath + 'common/_head.ejs', { page })`

根拠：
- `config/site.config.js`
- `src/**/*.html` 内の `include(...)`

### 3. EJSの出力演算子
- テキスト/属性に使う値は、既存コードに合わせる。
  - 既存では `common/_head.ejs` や `common/_header.ejs` で `<%- ... %>` を多用しています。
- タグ名などの“構文に影響する位置”では、既存に合わせて `<%= ... %>` を使います。

### 4. データ分離の実装方針
- 設計で与えられた文言（画像alt/説明文/ラベル等）を含むデータは、EJSの `<% ... %>` でオブジェクト化してから渡します。
- HTML側では、`<%- ... %>`/`<%= ... %>` で参照するだけにします。

NG（避ける）：
- HTMLタグ内に“新規の文言”をそのまま書き足す
- `<% ... %>` とHTML描画が細かく交互に混在する（ロジック過多）

OK（例の型）：
- `const pageIntro = { ... };` を用意して `include` に渡す（`thanks.html` の方針）

## CSS/Sass（SCSS）

### 1. 命名規則（BEM + プレフィックス）
- BEM前提で命名します。
- プレフィックス方針：
  - `p-`：大きいコンポーネント（セクション・カードなど複合要素）
  - `l-`：レイアウト
  - `c-`：小さいコンポーネント（ボタン・ラベルなど単体要素）
  - `u-`：ユーティリティ

### 2. ファイル配置
- Sassはコンポーネント単位で分割します。
  - 例：`src/assets/sass/components/_p-xxx.scss`
  - 例：`src/assets/sass/layouts/_l-xxx.scss`
  - 例：`src/assets/sass/utility/_u-xxx.scss`

### 3. 変換関数と余白・文字サイズ
- サイズ指定は原則 `rem()` / `maxrem()` を使用します。
- 余白は既存の書き方（指定順序の意図）に合わせます。

根拠：
- `src/assets/sass/global/_function.scss`

### 4. ブレークポイント
- レスポンシブは原則 `@include mq()` を使用します。
- モバイルファースト方針で、デスクトップ側は `@include mq()` 内に書きます。

根拠：
- `src/assets/sass/global/_breakpoints.scss`

### 5. SCSSの禁止事項
- `id` セレクタは使用しないでください。

## JavaScript

### 1. ファイル単位
- `src/assets/js/` に機能ごと（既存の `_*.js`）で分割し、必要に応じて `src/assets/js/main.js` からimportします。

根拠：
- `src/assets/js/main.js`

### 2. 初期化の書き方
- `DOMContentLoaded` で初期化し、対象要素が無い場合は早期returnしてください。

### 3. セレクタ運用
- JSで操作する要素は、可能な範囲で次のいずれかで管理します。
  - `data-*`（例：`data-fadein` のような用途）
  - `js-` 接頭辞の付いた `class`
  - `js-` 接頭辞の付いた `id`

既存コードの方針として `id="js-..."` が使われています（例：`js-header`, `js-drawer`）。この流儀に合わせてください。

### 4. アニメーション（GSAP）
- ScrollTrigger を使う場合は、既存のように `gsap.registerPlugin(ScrollTrigger)` と `matchMedia()` ベースの分岐を行います。

根拠：
- `src/assets/js/_top-message.js`

## PHP

### 1. 配列は `[]`
- PHPの配列は `array()` ではなく `[]` を使用してください。

### 2. データとHTMLの分離（可能な範囲）
- 既存の大きな資産（例：`mail.php`）は流用前提でも構いませんが、新規/改修を行う場合は
  - データ処理
  - 描画（HTML出力）
  をなるべく分離してください。

## 最終チェック（実行手順）

- ビルド/検証を通してからPR/納品してください。
  - `npm run build:only`
  - `npm run validate:html`
  - （または）`npm run validate:build`
